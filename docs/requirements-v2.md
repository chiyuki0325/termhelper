# 智能终端助手 需求分析文档 v2

## 1. 产品概述

### 1.1 产品定位

智能终端助手（termhelper）是一个 Linux 终端中的 TUI（终端用户界面）智能代理工具。用户以自然语言描述操作意图（如 `? 删除所有std开头的文件夹`），软件以全屏 TUI 界面弹出，调用大语言模型（LLM）将自然语言转换为可执行的 Shell 命令，辅以命令说明和安全警告，最终帮助用户安全、高效地完成终端操作。

### 1.2 目标用户

- 对 Linux 命令行不够熟悉的用户，希望用自然语言驱动终端操作
- 需要快速完成常见运维任务但不想记忆命令参数的开发者
- 希望降低误操作风险的终端使用者

### 1.3 核心价值

| 痛点 | 解决方案 |
|------|----------|
| 记不住命令和参数 | 自然语言输入，LLM 翻译为命令 |
| 不确定命令是否安全 | 自动风险评级，橙色警告危险操作 |
| 看不懂命令含义 | 自动生成命令分解说明 |
| 复杂交互式操作（如 sudo apt upgrade）需手动介入 | PTY 接管模式，LLM 代替用户与程序交互 |
| 重复调查系统环境 | 环境调查结果持久化，跨会话复用 |

## 2. 两种交互场景

原始需求描述了两个典型场景，它们的执行模型存在本质差异。以下给出形式化定义，作为后续功能需求的上下文。

### 场景 A：非交互式（普通）命令

**特征：** 命令执行过程不需要用户输入（密码、确认等）。一次性运行到结束。

**示例：** `rm -rf std*`、`ls -la`、`find . -name "*.log"`

**用户选项：**
- **运行** → spawn 子进程执行命令，执行完毕后展示退出码和输出
- **复制** → 复制到系统剪贴板，用户自行粘贴运行
- **修改** → 输入修改指示，LLM 重新生成命令

### 场景 B：交互式命令

**特征：** 命令执行过程中需要用户交互（sudo 密码、apt 确认提示等）。简单 spawn 无法完成，因为子进程的 stdin/stdout 与用户终端断连——普通用户无法直接向终端模拟打字。

**示例：** `sudo apt update && sudo apt upgrade`、`passwd`、`ssh user@host`

**用户选项：**
- **运行并接管** → spawn PTY，LLM 充当中间人监控输出并自动应答；程序完全接管交互
- **仅运行（复制）** → 将命令复制到剪贴板，由用户在自己的终端中粘贴并手动完成交互
- **修改** → 输入修改指示，LLM 重新生成命令

**关键设计决策：** 交互命令没有"普通 spawn"选项，因为 spawn 一个需要交互的子进程而用户终端未连接，命令将挂起或失败。若用户拒绝接管，唯一可行的执行路径是复制到剪贴板。

> 原始需求原文："如果选择运行和接管，大模型会 spawn 一个 pty 运行该程序……如果选择仅复制，程序将会把命令复制到剪贴板，让用户粘贴并运行。（普通用户并不可以直接往终端里模拟打字）"

## 3. 功能需求

### 3.1 FR-01 接入与启动

**触发方式：** 用户在 Shell 中输入 `? <自然语言描述>`。

**实现机制：**
- `termhelper` 为独立的仓颉二进制程序
- 安装时向用户的 Shell RC 文件（`~/.bashrc` / `~/.zshrc`）写入 alias：
  ```sh
  alias '?'='termhelper'
  ```
- 支持面需要覆盖 bash zsh fish nushell powershell
- 用户在终端输入 `? 删除所有std开头的文件夹` 后，Shell 将 `?` 解析为 alias，启动 `termhelper` 进程，后续文本作为命令行参数传入
- 接收到查询后，`termhelper` 以全屏 TUI 模式接管当前终端

**启动流程：**
1. Shell 解析 `?` → 启动 `termhelper <自然语言查询>`
2. termhelper 初始化 TUI 界面，显示加载状态
3. 加载已有环境上下文（如存在）
4. 构造 LLM 请求并发送
5. 展示结果界面

**验收标准：**
- 用户在任意目录下输入 `? <查询>` 均可启动 termhelper
- TUI 界面在 2s 内完成初始化并展示
- 退出 termhelper 后回到原 Shell，终端状态完好

### 3.2 FR-02 命令生成

**功能描述：** LLM 将自然语言查询转换为一条可执行的 Shell 命令。

**输入：** shell 类型 + 已存储的系统环境上下文（如有） + 用户的自然语言查询 
**输出：**
- 一条 Shell 命令
- 该命令的自然语言说明（命令用途、各关键参数的一句话解释）

**约束：**
- 每轮 LLM 调用只生成一条命令，不生成候选列表
- 若用户不满意，通过"修改"选项进入下一轮对话重新生成

**验收标准：**
- 生成的命令语法正确，可在目标 Shell 中执行
- 说明文字通俗易懂，覆盖命令用途及关键参数
- 每轮只返回一条命令

### 3.3 FR-03 环境调查（Agent 模式）

**功能描述：** 当 LLM 无法确定用户系统环境时，可发起环境调查——向用户申请执行诊断命令，获取系统信息后再生成最终命令。

**典型场景：**
```
用户: ? 更新系统
LLM:  需要了解系统包管理器，申请执行: cat /etc/os-release
用户: 同意
LLM:  (获取到 Debian 信息) 生成命令: sudo apt update && sudo apt upgrade
      储存结论: "用户正在使用 Debian Linux"
```

**调查流程：**
1. LLM 判断当前查询需要环境信息
2. LLM 生成诊断命令并请求用户授权
3. 用户同意后，执行诊断命令并获取输出
4. LLM 基于诊断结果生成最终命令
5. 环境调查结果持久化存储，后续会话可载入复用

**验收标准：**
- LLM 可主动发起环境调查请求
- 诊断命令需经用户授权方可执行
- 调查结果正确存储并可被后续会话加载

### 3.4 FR-04 命令说明生成

**功能描述：** 为生成的命令提供人类可读的说明。

**说明内容要求：**
- 命令整体用途概述（一句话）
- 各关键参数/子命令的作用解释（一句话）
- 颜色编码：正常信息默认色，危险警告用橙色

**示例：**
```
命令: rm -rf std*
说明:
  rm   — 删除文件或目录
  -r   — 递归删除目录及其内容
  -f   — 强制删除，不提示确认
  ⚠ 该命令具有危险性，删除的文件无法恢复 [橙色]
```

### 3.5 FR-05 安全分析

**功能描述：** 自动分析命令的风险等级并给予相应提示。

**风险等级定义：**

| 等级 | 触发条件 | 提示方式 |
|------|----------|----------|
| 安全 | 纯读取操作（ls, cat, echo 等） | 无额外警告 |
| 注意 | 文件写入/修改操作 | 温和提示 |
| 危险 | 不可逆删除（rm -rf）、格式化（mkfs）、磁盘操作（dd） | 橙色醒目警告 |
| 提权 | 需要 sudo/root 权限 | 警告需管理员权限，提示接管模式 |

**验收标准：**
- 至少识别删除类、格式化类、提权类三类危险命令
- 危险警告以橙色渲染
- 危险命令在执行前需额外确认

### 3.6 FR-06 用户交互选项

**功能描述：** 命令生成后，根据命令类型向用户展示对应的交互选项。

**场景 A（非交互式命令）选项：**

| 选项 | 行为 |
|------|------|
| **运行** | spawn 子 Shell 执行命令，完成后展示退出码和输出 |
| **复制** | 将命令复制到系统剪贴板，由用户自行粘贴运行 |
| **修改** | 输入进一步的修改指示，进入下一轮 LLM 对话重新生成 |

**场景 B（交互式命令）选项：**

| 选项 | 行为 |
|------|------|
| **运行并接管** | spawn PTY，LLM 监控输出并自动应答，完全接管终端交互 |
| **仅运行（复制）** | 将命令复制到系统剪贴板，用户在自己的终端中手动交互执行 |
| **修改** | 输入进一步的修改指示，进入下一轮 LLM 对话重新生成 |

**场景判定规则：**
- 由 LLM 在生成命令时判定命令属于场景 A 还是场景 B
- 判定依据：命令是否包含需要用户交互的组件（sudo、ssh、passwd、包管理器的确认提示等）
- 判定结果随命令一起返回，驱动后续交互选项的展示

**验收标准：**
- 两种场景的选项菜单正确展示
- 选项选择方式便捷（数字/字母快捷键）
- 场景判定准确率满足可用性要求

### 3.7 FR-07 命令执行

**功能描述：** 在用户确认后，根据场景类型和用户选择执行命令。

**模式一：普通执行（对应场景 A 的"运行"）**
- 实现方式：spawn 一个子 Shell 进程执行命令
- 执行完成后展示退出码和输出（stdout + stderr）
- 适用条件：仅当命令被 LLM 判定为非交互式，且用户选择"运行"

**模式二：PTY 接管执行（对应场景 B 的"运行并接管"）**
- 实现方式：spawn 一个 PTY（伪终端），LLM 监控程序输出并自动进行交互操作
- LLM 充当中间人，读取程序 stdout，识别提示符（密码提示 `[sudo] password:`、确认提示 `[Y/n]` 等），自动发送应答输入
- 适用条件：仅当命令被 LLM 判定为交互式，且用户选择"运行并接管"
- 执行过程中用户可随时中断（如 Ctrl+C）

**模式三：复制（两种场景通用）**
- 将命令文本复制到系统剪贴板
- 场景 A 的"复制"和场景 B 的"仅复制"均走此路径
- 复制后展示已复制提示，用户自行在终端粘贴执行

**重要说明：**
- 场景 B 的命令不存在"普通 spawn 然后让用户手动交互"的路径，因为用户终端与子进程的 stdin/stdout 不连通——普通用户无法直接向终端模拟打字
- 场景 A 的命令不存在 PTY 接管路径，因为没有交互需求

**验收标准：**
- 普通命令可正常 spawn 执行并展示结果
- PTY 模式下 LLM 能正确识别常见程序提示并自动应答
- 执行过程可被用户中断（Ctrl+C / ESC）
- 复制功能在 xclip 和 wl-clipboard 环境下均可工作

### 3.8 FR-08 环境上下文管理

**功能描述：** 存储和加载系统环境调查结果，避免重复调查。

**存储内容：**
- 操作系统及发行版信息
- 包管理器类型
- 已安装的关键工具及版本
- 其他 LLM 认为有复用价值的调查结论

**存储方式：** 本地 JSON 文件持久化（`~/.config/termhelper/context.json`）

**生命周期：**
- 首次使用：环境上下文为空，LLM 按需调查
- 后续使用：自动加载已有上下文，LLM 判断是否仍有效
- 更新：系统环境变化时 LLM 可发起重新调查

**验收标准：**
- 环境上下文可持久化存储
- 新会话启动时自动加载已有上下文
- 上下文变化时可更新

### 3.9 FR-09 LLM 对话管理

**功能描述：** 管理多轮对话上下文，支持命令修改和追问。

**支持场景：**
- 用户对生成的命令不满意，输入修改指示
- LLM 需要澄清用户意图时发起提问
- 环境调查时为诊断命令请求授权

**上下文管理：**
- 每个会话维护对话历史
- 对话历史随每次交互追加
- 单个会话内 LLM 可引用之前的所有交互

**交互说明：**
- "修改"选项选中后，用户在 TUI 内的文本输入区域输入修改指示（自由文本），LLM 基于对话历史和修改指示重新生成命令
- 修改流程复用已有的 LLM 对话管理能力，不启动新的 termhelper 进程

**验收标准：**
- 支持多轮对话修改命令
- 对话历史在会话内保持连贯

### 3.10 FR-10 卸载与清理

**功能描述：** 提供卸载机制，清理 termhelper 写入的 alias 和配置文件。

**清理范围：**
- Shell RC 文件中的 `alias '?'='termhelper'` 行
- `~/.config/termhelper/` 目录下的上下文文件和配置

**验收标准：**
- 卸载后 `?` 不再触发 termhelper
- 用户数据（上下文文件）按用户选择保留或删除

## 4. 非功能需求

### 4.1 NFR-01 性能

| 指标 | 要求 | 备注 |
|------|------|------|
| 冷启动时间 | ≤ 2s | 从用户输入 `?` 到 TUI 界面就绪 |
| LLM 响应延迟 | 取决于 API | 需有加载状态提示（TUI 内 spinner/进度指示） |
| 环境上下文加载 | ≤ 100ms | JSON 文件读取 |

### 4.2 NFR-02 可靠性

- LLM 调用失败时给出友好错误提示，不崩溃，不退出 TUI（允许用户重试）
- 命令执行超时时允许用户中断
- 网络异常时提供重试机制

### 4.3 NFR-03 安全性

- 危险命令必须经用户二次确认
- 诊断命令必须经用户授权方可执行
- 不在用户不知情的情况下执行任何 Shell 命令
- 环境上下文文件中不存储敏感信息（如密码、Token）
- LLM API Key 安全存储（环境变量或配置文件，不硬编码）

### 4.4 NFR-04 可用性

- 命令说明使用用户可理解的自然语言，避免术语堆砌
- 颜色编码清晰一致：危险=橙色，正常=默认色，成功=绿色
- 错误信息包含可操作的解决建议
- TUI 导航符合终端用户习惯（键盘驱动，快捷键明确）

### 4.5 NFR-05 可扩展性

- LLM 后端可替换（支持不同模型提供商）
- 命令安全规则可扩展（新增危险模式识别）
- 支持新增交互选项类型

### 5.1 模块划分

```
┌──────────────────────────────────────────────┐
│              Shell 层（用户终端）              │
│   ? <查询> ──alias──→ termhelper 二进制       │
└──────────────────┬───────────────────────────┘
                   │
┌──────────────────▼───────────────────────────┐
│              TUI 层                           │
│  全屏终端界面：加载态 → 结果显示 → 交互选项    │
└──────────────────┬───────────────────────────┘
                   │
┌──────────────────▼───────────────────────────┐
│              核心引擎                          │
│  ┌──────────┐ ┌──────────┐ ┌──────────────┐  │
│  │ 对话管理  │ │ 上下文管理│ │ 安全分析器   │  │
│  └──────────┘ └──────────┘ └──────────────┘  │
│  ┌──────────┐ ┌──────────┐ ┌──────────────┐  │
│  │ LLM客户端 │ │ 命令执行器│ │ 场景判定器   │  │
│  └──────────┘ └──────────┘ └──────────────┘  │
└──────────────────┬───────────────────────────┘
                   │
┌──────────────────▼───────────────────────────┐
│              基础设施层                        │
│  ┌──────────┐ ┌──────────┐ ┌──────────────┐  │
│  │ 剪贴板   │ │ 文件存储  │ │ Shell/PTY    │  │
│  └──────────┘ └──────────┘ └──────────────┘  │
└──────────────────────────────────────────────┘
```

### 5.2 核心数据流

```
用户输入 ?查询
    │
    ▼
Shell alias 解析 → 启动 termhelper 进程（传入查询文本）
    │
    ▼
TUI 初始化（全屏接管终端）
    │
    ▼
加载环境上下文 ──→ 构造 LLM 请求（含 system prompt + 历史上下文）
    │
    ▼
LLM API 调用 ──→ 解析 LLM 响应
    │
    ├── LLM 请求环境调查？
    │       │
    │       ▼
    │   展示诊断命令 → 等待用户授权
    │       │
    │       ├── 同意 → 执行诊断命令 → 回传结果 → 新一轮 LLM 调用
    │       └── 拒绝 → 提示无法继续，返回修改/退出
    │
    ├── LLM 无法理解？
    │       │
    │       ▼
    │   展示澄清提问 → 等待用户输入 → 新一轮 LLM 调用
    │
    └── LLM 生成命令 + 场景判定（A: 非交互式 / B: 交互式）
            │
            ▼
       安全分析 + 渲染显示（命令 + 说明 + 警告）
            │
            ▼
    ┌─── 场景 A 选项 ───┐     ┌─── 场景 B 选项 ───┐
    │ [运行] [复制] [修改] │     │ [运行并接管] [仅运行(复制)] [修改] │
    └─────────┬──────────┘     └──────────┬──────────┘
              │                           │
    ┌─────────┼─────────┐       ┌─────────┼─────────┐
    ▼         ▼         ▼       ▼         ▼         ▼
  spawn    复制到    新一轮   PTY     复制到    新一轮
  执行     剪贴板   LLM调用  接管执行  剪贴板   LLM调用
    │                           │
    ▼                           ▼
 展示结果                  LLM自动交互
 (退出码+输出)             直到命令完成
```

## 6. 用户故事

### US-01 新手删除文件
> 作为不熟悉 Linux 命令的用户，我希望输入"删除所有 tmp 开头的文件"，系统能告诉我该用什么命令、解释命令含义、并警告风险，让我安全地完成操作。命令不涉及交互，我选择"运行"直接执行。

### US-02 开发者更新系统
> 作为开发者，我希望输入"更新系统"后，系统能自动识别我的 Linux 发行版和包管理器，生成正确的更新命令。由于命令需要 sudo，系统判定为交互式命令，展示"运行并接管"选项——我选择后 LLM 自动处理 sudo 密码输入和 apt 确认提示。

### US-03 不满意初版命令
> 作为用户，当 LLM 生成的命令不符合预期时，我希望能够输入修改指示让它重新生成，而不是退出 termhelper 再手动重写整个查询。

### US-04 只想复制命令
> 作为有经验的用户，我不需要软件帮我执行命令，只想快速拿到命令复制到剪贴板，自己决定何时以及如何执行。

### US-05 不想让 LLM 接管终端
> 作为谨慎的用户，当生成的命令需要交互（如 sudo）时，我可能不想让 LLM 完全接管我的终端。我选择"仅运行（复制）"，将命令复制到剪贴板，在自己的终端中手动完成交互操作。

## 7. 术语表

| 术语 | 定义 |
|------|------|
| 自然语言查询 | 用户在 `?` 后输入的日常语言描述，如"删除所有 std 开头的文件夹" |
| TUI | 终端用户界面，termhelper 以全屏模式接管终端进行交互展示 |
| 诊断命令 | LLM 为调查系统环境而请求执行的探测性命令，如 `cat /etc/os-release` |
| 环境上下文 | 经调查获得的系统环境信息集合，持久化存储供后续复用 |
| 场景 A / 非交互式命令 | 执行过程不需要用户交互的命令（如 `rm`、`ls`），可直接 spawn 执行 |
| 场景 B / 交互式命令 | 执行过程中需要用户交互的命令（如 `sudo apt upgrade`），必须通过 PTY 接管或降级复制 |
| PTY 接管 | 通过伪终端让 LLM 监控程序输出并自动应答，模拟人工交互；对应场景 B 的"运行并接管" |
| 普通执行 | spawn 子进程执行命令；仅适用于场景 A |
| Shell RC alias | 在 `~/.bashrc` / `~/.zshrc` 中写入 `alias '?'='termhelper'`，使 `?` 被 Shell 解析为 termhelper 调用 |
| 安全分析 | 对生成命令进行风险评级并产生对应警告提示的过程 |

## 8. 待澄清问题

1. **LLM 提供商选型：** 使用哪个大模型 API？（OpenAI / Anthropic / 本地模型）API 配置方式如何？
2. **多 Shell 支持：** 是否需要支持 bash 以外的 Shell（zsh, fish）？安装时需向哪些 RC 文件写入 alias？
3. **TUI 框架选型：** 仓颉生态中可用的 TUI 库有哪些？是否满足全屏接管、键盘导航、颜色渲染等需求？
4. **会话历史：** 是否需要保存跨启动会话的对话历史，还是仅保留环境上下文？
5. **命令执行工作目录：** 命令是在用户触发 `?` 时所在目录执行，还是 termhelper 内部维护工作目录？
6. **并发安全：** 是否允许多个终端窗口同时使用 termhelper？若允许，环境上下文读写如何同步？
7. **剪贴板降级策略：** xclip/wl-clipboard 均不可用时，是否降级为仅显示命令让用户手动复制？
8. **命令执行超时：** 是否需要超时限制？默认超时时间是多少？
9. **PTY 接管的白名单/黑名单：** 接管模式下 LLM 自动应答的边界是什么——是否只应答已知的提示模式（`[sudo] password`、`[Y/n]`），还是允许 LLM 自由决策？若遇到未知提示如何处理？
10. **国际化：** 是否需要支持中英文以外的语言？LLM 返回的命令说明语言如何控制？
