# 智能终端助手 需求分析文档 v5

## 1. 产品概述

### 1.1 产品定位

智能终端助手（termhelper）是一个 Linux 终端中的 TUI（终端用户界面）智能代理工具。用户以自然语言描述操作意图（如 `?? 删除所有std开头的文件夹`），软件以全屏 TUI 界面弹出，调用大语言模型（LLM）将自然语言转换为可执行的 Shell 命令，辅以命令说明和安全警告，最终帮助用户安全、高效地完成终端操作。

### 1.2 目标用户

- 对 Linux 命令行不够熟悉的用户，希望用自然语言驱动终端操作
- 需要快速完成常见运维任务但不想记忆命令参数的开发者
- 希望降低误操作风险的终端使用者

### 1.3 核心价值

| 痛点 | 解决方案 |
|------|----------|
| 记不住命令和参数 | 自然语言输入，LLM 翻译为命令 |
| 不确定命令是否安全 | 自动风险评级，橙色警告危险操作 |
| 看不懂命令含义 | 自动生成命令分解说明 |
| 复杂交互式操作（如 sudo apt upgrade）需手动介入 | PTY 接管模式，LLM 代替用户与程序交互 |
| 重复调查系统环境 | 环境调查结果持久化，跨会话复用 |

## 2. 两种交互场景

原始需求描述了两个典型场景，它们的执行模型存在本质差异——区别在于命令是否需要与用户交互（密码、确认提示等）。LLM 通过结构化响应中的 `interactive` 字段（布尔值）判定。

### 非交互式命令（`interactive: false`）

**特征：** 命令执行过程不需要用户输入（密码、确认等）。一次性运行到结束。

**示例：** `rm -rf std*`、`ls -la`、`find . -name "*.log"`

**用户选项：**
- **运行** → spawn 子进程执行命令，执行完毕后展示退出码和输出
- **复制** → 复制到系统剪贴板，用户自行粘贴运行
- **修改** → 输入修改指示，LLM 重新生成命令

### 交互式命令（`interactive: true`）

**特征：** 命令执行过程中需要用户交互（sudo 密码、apt 确认提示等）。简单 spawn 无法完成，因为子进程的 stdin/stdout 与用户终端断连——普通用户无法直接向终端模拟打字。

**示例：** `sudo apt update && sudo apt upgrade`、`passwd`、`ssh user@host`

**用户选项：**
- **运行并接管** → spawn PTY，LLM 监控输出并自动应答（权限模型见 FR-07）；程序完全接管交互
- **仅复制** → 将命令复制到系统剪贴板，用户在自己的终端中手动交互执行
- **修改** → 输入修改指示，LLM 重新生成命令

**关键设计决策：** 交互命令没有"普通 spawn"选项，因为 spawn 一个需要交互的子进程而用户终端未连接，命令将挂起或失败。若用户拒绝接管，唯一可行的执行路径是复制到剪贴板。

> 原始需求原文："如果选择运行和接管，大模型会 spawn 一个 pty 运行该程序……如果选择仅运行，程序将会把命令复制到剪贴板，让用户粘贴并运行。（普通用户并不可以直接往终端里模拟打字）"

## 3. 功能需求

### 3.1 FR-01 接入与启动

**触发方式：** 用户在 Shell 中输入 `?? <自然语言描述>`。

**触发词选择说明：** 原始需求使用 `?`，但 `?` 是 Shell 单字符通配符（glob），当前目录存在单字符文件名时会被展开而导致 alias 失效。`??` 更接近原意，且作为命令首词时 Shell 的 alias 展开优先于 pathname 展开，冲突概率显著降低。若用户环境中 `??` 仍偶发 glob 冲突，可在输入时加反斜杠转义（`\?\?`）。

**安装机制：**
- 安装时向用户的 Shell RC 文件写入命令定义。各 Shell 语法差异如下：

| Shell | RC 文件 | 命令定义 |
|-------|---------|----------|
| bash | `~/.bashrc` | `alias '??'='termhelper'` |
| zsh | `~/.zshrc` | `alias '??'='termhelper'` |
| fish | `~/.config/fish/config.fish` | `alias '??' termhelper`（fish 语法无 `=`） |
| nushell | `~/.config/nushell/config.nu` | `def "??" [ ...args ] { termhelper ...$args }`（nushell 的 alias 不支持参数传递，改用 `def` 定义命令；具体语法留待实现阶段验证） |

> **注意：** nushell 的 alias 机制不支持参数传递，因此使用 `def` 命令定义。`def` 的参数展开语法（`...$args`）需在实现阶段验证具体版本兼容性。powershell（pwsh）不在 v1 支持范围内。

- 用户在终端输入 `?? 删除所有std开头的文件夹` 后，Shell 将 `??` 解析为 termhelper 调用，后续文本作为参数传入
- 接收到查询后，`termhelper` 以全屏 TUI 模式接管当前终端

**启动流程：**
1. Shell 解析 `??` → 启动 `termhelper <自然语言查询>`
2. termhelper 初始化 TUI 界面，显示加载状态
3. 加载已有环境上下文（如存在）
4. 构造 LLM 请求并发送
5. 展示结果界面

**验收标准：**
- 用户在任意目录下输入 `?? <查询>` 均可启动 termhelper
- TUI 界面在 2s 内完成初始化并展示
- 退出 termhelper 后回到原 Shell，终端状态完好

### 3.2 FR-02 命令生成

**功能描述：** LLM 将自然语言查询转换为一条可执行的 Shell 命令。

**输入：** shell 类型 + 已存储的系统环境上下文（如有） + 用户的自然语言查询 + 当前对话历史（单会话内）

**LLM 结构化输出格式（JSON Schema）：**

LLM 的每次响应必须为以下三种类型之一，以 JSON 格式返回：

```jsonc
// 类型一：生成最终命令
{
  "type": "command",
	"data": {
  "command": "rm -rf std*",
  "explanation": {
    "summary": "删除所有 std 开头的文件夹",
    "breakdown": [
      {"component": "rm", "explanation": "删除文件或目录"},
      {"component": "-r", "explanation": "递归删除目录及其内容"},
      {"component": "-f", "explanation": "强制删除，不提示确认"},
      {"component": "std*", "explanation": "匹配所有 std 开头的文件/文件夹"}
    ]
  },
  "interactive": false,
  "safety": {
    "level": "danger",
    "warnings": ["该命令具有危险性，删除的文件无法恢复"]
  }
	}
}

// 类型二：请求环境调查
{
  "type": "investigate",
	"data": {
  "reason": "需要了解系统使用的包管理器以生成正确的更新命令",
  "commands": [
		{"command": "cat /etc/os-release", "rationale": "获取 Linux 发行版信息"},
		{"command": "which apt || which dnf || which pacman"], "rationale": "探测可用的包管理器"}
  ]
	}
}

// 类型三：向用户澄清意图
{
  "type": "clarify",
	"data": {
  "question": "您想删除的是文件名以 std 开头的文件，还是文件名包含 std 的文件？"
	}
}
```

**`interactive` 字段说明：**
- `false`：命令执行过程不需要用户交互，走普通 spawn 路径（见 FR-07 模式一）
- `true`：命令执行过程中需要用户交互（sudo 密码、包管理器确认、SSH 密码等），走 PTY 接管路径或降级为复制（见 FR-07 模式二/三）
- 由 LLM 根据命令内容判定：是否包含 sudo、ssh、passwd、包管理器的交互式确认提示等

**约束：**
- 每轮 LLM 调用只生成一条命令，不生成候选列表
- 若用户不满意，通过"修改"选项进入下一轮对话重新生成
- JSON 解析失败时，向用户展示原始响应并提示重试

**验收标准：**
- 生成的命令语法正确，可在目标 Shell 中执行
- 说明文字通俗易懂，覆盖命令用途及关键参数
- 每轮只返回一条命令
- JSON 响应的三个类型均可正确解析

### 3.3 FR-03 环境调查（Agent 模式）

**功能描述：** 当 LLM 无法确定用户系统环境时，可发起环境调查——通过结构化 JSON 响应（`type: "investigate"`）向用户申请执行诊断命令，获取系统信息后再生成最终命令。

**典型场景：**
```
用户: ?? 更新系统
LLM:  响应 {type: "investigate" ... }
TUI:   展示诊断命令列表及理由，等待用户授权
用户: 同意
系统:  执行 cat /etc/os-release，获取 Debian 信息
LLM:  （新一轮调用，含诊断结果）响应 {type: "command", command: "sudo apt update && sudo apt upgrade", ...}
      同时储存结论: "用户正在使用 Debian Linux，包管理器为 apt"
```

**交互协议：**
1. LLM 返回 `type: "investigate"` 的 JSON 响应
2. TUI 展示调查理由和待执行的诊断命令列表，每条命令附说明
3. 用户逐条或批量授权诊断命令
4. 被授权的命令 spawn 执行（诊断命令默认非交互式，走普通 spawn），收集输出
5. 将诊断命令及输出作为新消息追加到对话上下文，发起新一轮 LLM 调用
6. LLM 基于诊断结果生成最终命令，并在响应中包含环境上下文更新
7. 环境调查结果持久化存储（见 FR-08），后续会话可载入复用

**验收标准：**
- LLM 可主动发起环境调查请求（通过 `type: "investigate"` 响应）
- 诊断命令需经用户授权方可执行
- 调查结果正确存储并可被后续会话加载
- 用户可拒绝部分或全部诊断命令

### 3.4 FR-04 命令说明生成

**功能描述：** 为生成的命令提供人类可读的说明。说明内容来自 LLM 结构化响应中的 `explanation` 字段。

**说明内容要求：**
- 命令整体用途概述（一句话，`explanation.summary`）
- 各关键参数/子命令的作用解释（一句话，`explanation.breakdown[]`）
- 颜色编码：正常信息默认色，危险警告用橙色

**示例：**
```
命令: rm -rf std*
说明:
  rm   — 删除文件或目录
  -r   — 递归删除目录及其内容
  -f   — 强制删除，不提示确认
  ⚠ 该命令具有危险性，删除的文件无法恢复 [橙色]
```

### 3.5 FR-05 安全分析

**功能描述：** 自动分析命令的风险等级并给予相应提示。风险等级由 LLM 在结构化响应中判定，TUI 根据等级渲染对应警告。

**风险等级定义：**

| 等级 | 触发条件 | 提示方式 |
|------|----------|----------|
| 安全 | 纯读取操作（ls, cat, echo 等） | 无额外警告 |
| 注意 | 文件写入/修改操作 | 温和提示 |
| 危险 | 不可逆删除（rm -rf）、格式化（mkfs）、磁盘操作（dd） | 橙色醒目警告 |
| 提权 | 需要 sudo/root 权限 | 警告需管理员权限，提示接管模式 |

**验收标准：**
- 至少识别删除类、格式化类、提权类三类危险命令
- 危险警告以橙色渲染
- 危险/提权命令在执行前需额外确认

### 3.6 FR-06 用户交互选项

**功能描述：** 命令生成后，根据 LLM 返回的 `interactive` 字段（`true` / `false`）向用户展示对应的交互选项。

**非交互式命令（`interactive: false`）选项：**

| 选项 | 行为 |
|------|------|
| **运行** | spawn 子 Shell 执行命令，完成后展示退出码和输出 |
| **复制** | 将命令复制到系统剪贴板，由用户自行粘贴运行 |
| **修改** | 输入进一步的修改指示，进入下一轮 LLM 对话重新生成 |

**交互式命令（`interactive: true`）选项：**

| 选项 | 行为 |
|------|------|
| **运行并接管** | spawn PTY，进入接管模式（权限模型见 FR-07） |
| **仅复制** | 将命令复制到系统剪贴板，用户在自己的终端中手动交互执行 |
| **修改** | 输入进一步的修改指示，进入下一轮 LLM 对话重新生成 |

**场景判定：**
- 由 LLM 在结构化响应中的 `interactive` 字段判定（`true` / `false`）
- 判定依据：命令是否包含需要用户交互的组件（sudo、ssh、passwd、包管理器的确认提示等）

**验收标准：**
- 两种场景的选项菜单正确展示
- 各选项（运行/复制/修改 及 运行并接管/仅复制/修改）正常响应

### 3.7 FR-07 命令执行

**功能描述：** 在用户确认后，根据 `interactive` 字段和用户选择执行命令。命令在用户触发 `??` 时所在的当前工作目录下执行。

**模式一：普通执行（对应 `interactive: false` 的"运行"）**
- 实现方式：spawn 一个子 Shell 进程执行命令
- 执行完成后展示退出码和输出（stdout + stderr）
- 适用条件：仅当 `interactive` 为 `false`，且用户选择"运行"

**模式二：PTY 接管执行（对应 `interactive: true` 的"运行并接管"）**
- 实现方式：spawn 一个 PTY（伪终端），LLM 监控程序输出并自动进行交互操作
- 权限模型：

  ```
  默认模式（逐次确认）：
    PTY 有新输出 → TUI 展示输出
    → LLM 分析输出，决定需要向 PTY 写入什么内容
    → TUI 展示 LLM 计划写入的内容及理由
    → 用户确认或拒绝
    → 确认后写入 PTY，继续监控

  全自动模式（用户主动切换）：
    用户切换至全自动模式
    → LLM 完全自主决策，无需逐次确认
    → 用户仍可随时中断整个命令
    → 用户可切回默认模式
  ```

- LLM 充当中间人：每轮收到 PTY 新输出后，LLM 决策下一步操作（发送密码、确认 [Y/n]、输入其他内容，或判断命令已完成）
- 适用条件：仅当 `interactive` 为 `true`，且用户选择"运行并接管"

**模式三：复制（两种场景通用）**
- 将命令文本复制到系统剪贴板
- `interactive: false` 的"复制"和 `interactive: true` 的"仅复制"均走此路径
- 复制后展示已复制提示，用户自行在终端粘贴执行
- 剪贴板工具选择：根据 `WAYLAND_DISPLAY` 环境变量检测 Wayland 会话（使用 wl-clipboard），根据 `XDG_SESSION_TYPE` 检测 X11 会话（使用 xclip）
- 降级策略：若对应工具未安装，仅将命令展示在 TUI 界面中，并提示用户安装对应工具（Wayland → `wl-clipboard`，X11 → `xclip`）

**重要说明：**
- `interactive: true` 的命令不存在"普通 spawn 然后让用户手动交互"的路径，因为用户终端与子进程的 stdin/stdout 不连通——普通用户无法直接向终端模拟打字
- `interactive: false` 的命令不存在 PTY 接管路径，因为没有交互需求

**验收标准：**
- 普通命令可正常 spawn 执行并展示结果
- PTY 模式默认需要用户逐次确认写入操作
- 全自动模式可正确切换和退出
- 执行过程可被用户中断
- 复制功能在 xclip 和 wl-clipboard 环境下均可工作
- 剪贴板工具缺失时正确降级为展示命令并提示安装

### 3.8 FR-08 环境上下文管理

**功能描述：** 存储和加载系统环境调查结果，避免重复调查。

**存储内容：**
- 操作系统及发行版信息
- 包管理器类型
- 已安装的关键工具及版本
- 其他 LLM 认为有复用价值的调查结论

**存储方式：** 本地 JSON 文件持久化（`~/.config/termhelper/context.json`）

**生命周期：**
- 首次使用：程序做轻量本地探测并写入 facts（OS / distro / 包管理器列表 / 常用工具）；涉及非标准工具、发行版相关命令、包管理器、服务管理器等具体操作时，LLM 必须先从已知 facts 或当前会话调查结果确认所需工具存在，再给命令。
- 后续使用：自动加载已有上下文，作为 LLM 请求的一部分发送
- 更新：系统环境变化时 LLM 可发起重新调查；每次调查完成后通过 `factEdits` 新增、修改或删除 `context.json` 中的 facts。

**注意：** 环境上下文跨会话持久化，但 Shell 不写入 `context.json`，而是在构造 prompt 时动态注入；对话历史不跨会话保留（见 FR-09）。

**验收标准：**
- 环境上下文可持久化存储
- 新会话启动时自动加载已有上下文
- 上下文变化时可更新

### 3.9 FR-09 LLM 对话管理

**功能描述：** 管理会话内的多轮对话上下文，支持命令修改和追问。

**会话定义：** 一次 `??` 触发、TUI 启动到 TUI 退出为一个完整会话。退出 TUI 后对话历史即丢弃，不持久化。下次 `??` 启动时仅载入环境上下文（FR-08），对话从零开始。

**支持场景：**
- 用户对生成的命令不满意，通过"修改"选项输入修改指示
- LLM 需要澄清用户意图时发起提问（`type: "clarify"`）
- 环境调查时为诊断命令请求授权（`type: "investigate"`）

**上下文管理：**
- 每个会话维护一个消息列表（system prompt + 环境上下文 + 对话历史）
- 对话历史随每次交互追加（用户查询、LLM 响应、诊断命令结果、修改指示等均追加）
- 会话内 LLM 可引用之前的所有交互

**"修改"流程：**
- 用户选择"修改"后，在 TUI 内的文本输入区域输入修改指示（自由文本）
- 修改指示作为新消息追加到对话历史
- 发起新一轮 LLM 调用，LLM 基于完整对话历史重新生成命令
- 不启动新的 termhelper 进程

**验收标准：**
- 支持会话内多轮对话修改命令
- 对话历史在会话内保持连贯
- 退出 TUI 后对话历史不保留

### 3.10 FR-10 卸载与清理

> **设计说明：** 此功能为原始需求之外的新增建议。原始需求未提及卸载机制，但作为面向终端用户的工具，提供干净的配置清理路径属于合理预期。程序本身的卸载由包管理器负责，Shell RC 文件的编辑由用户自行处理。

**功能描述：** 提供清理命令，移除 termhelper 写入的命令定义片段和配置文件。

**清理范围：**
- Shell RC 文件中 termhelper 写入的 `alias '??'='termhelper'` 行（及各 Shell 对应语法）——仅删除 termhelper 安装时追加的 alias/def 行，不修改用户的其他配置
- `~/.config/termhelper/` 目录下的上下文文件和配置

**不在清理范围内：**
- termhelper 程序二进制（由包管理器负责）
- Shell RC 文件中与 termhelper 无关的配置（由用户自行管理）

**验收标准：**
- 清理后 `??` 不再触发 termhelper
- 用户数据（上下文文件）按用户选择保留或删除
- 不影响 Shell RC 文件中 termhelper 命令定义以外的其他配置

## 4. 非功能需求

### 4.1 NFR-01 性能

| 指标 | 要求 | 备注 |
|------|------|------|
| 冷启动时间 | ≤ 2s | 从用户输入 `??` 到 TUI 界面就绪 |
| LLM 响应延迟 | 取决于 API | 需有加载状态提示（TUI 内 spinner/进度指示） |
| 环境上下文加载 | ≤ 100ms | JSON 文件读取 |

### 4.2 NFR-02 可靠性

- LLM 调用失败时给出友好错误提示，不崩溃，不退出 TUI（允许用户重试）
- LLM 返回的 JSON 无法解析时，展示原始响应并提示重试（不崩溃）
- 命令执行超时时允许用户中断（超时时间留待系统设计阶段确定）
- 网络异常时提供重试机制

### 4.3 NFR-03 安全性

- 危险命令必须经用户二次确认
- 诊断命令必须经用户授权方可执行
- 不在用户不知情的情况下执行任何 Shell 命令
- PTY 接管模式默认需要逐次确认（"默认模式"），防止 LLM 误操作
- 环境上下文文件中不存储敏感信息（如密码、Token）
- LLM API Key 安全存储（环境变量或配置文件，不硬编码）

### 4.4 NFR-04 可用性

- 命令说明使用用户可理解的自然语言，避免术语堆砌
- 颜色编码清晰一致：危险=橙色，正常=默认色，成功=绿色
- 错误信息包含可操作的解决建议
- TUI 导航符合终端用户习惯（键盘驱动）

### 4.5 NFR-05 可扩展性

- LLM 后端通过抽象接口支持多提供商替换（见 5.3 节）
- 命令安全规则可扩展（新增危险模式识别）
- 支持新增交互选项类型

### 5.2 核心数据流

```
用户输入 ??查询
    │
    ▼
Shell alias/def 解析 → 启动 termhelper 进程（传入查询文本）
    │
    ▼
TUI 初始化（全屏接管终端）
    │
    ▼
加载环境上下文 ──→ 构造 LLM 请求（system prompt + 环境上下文 + 用户查询）
    │
    ▼
LLM API 调用 ──→ 解析 LLM JSON 响应
    │
    ├── type: "investigate"（环境调查请求）？
    │       │
    │       ▼
    │   展示诊断命令列表 + 理由 → 等待用户逐条/批量授权
    │       │
    │       ├── 用户授权 → spawn 执行诊断命令 → 收集输出
    │       │      │
    │       │      ▼
    │       │   诊断结果追加到对话历史 → 新一轮 LLM 调用
    │       │
    │       └── 用户拒绝 → 提示无法继续，回退到修改/退出
    │
    ├── type: "clarify"（澄清提问）？
    │       │
    │       ▼
    │   展示 LLM 的提问 → 等待用户输入 → 新一轮 LLM 调用
    │
    └── type: "command"（生成命令）？
            │
            ▼
       安全分析 + 渲染显示（命令 + 说明 + 警告）
       interactive 字段判定: false → 非交互式 / true → 交互式
            │
            ▼
    ┌── interactive: false ──┐  ┌── interactive: true ──┐
    │ 运行 | 复制 | 修改       │  │ 运行并接管 | 仅复制 | 修改│
    └─────────┬──────────────┘  └──────────┬─────────────┘
              │                           │
    ┌─────────┼─────────┐       ┌─────────┼─────────┐
    ▼         ▼         ▼       ▼         ▼         ▼
  spawn    复制到    新一轮   PTY     复制到    新一轮
  执行     剪贴板   LLM调用  接管执行  剪贴板   LLM调用
    │                           │
    ▼                           ▼
 展示结果              ┌─ 默认模式：LLM建议写入→用户逐次确认
 (退出码+输出)          │─ 全自动模式(用户切换)：LLM完全自主
                       │─ 用户可随时中断
                       └─ 命令完成或中断后展示摘要
```

### 5.3 技术栈

| 层 | 技术选型 | 说明 |
|-----|---------|------|
| 编程语言 | 仓颉（Cangjie） 1.1.0 | 项目约束 |

**LLM 客户端设计：**

```
┌──────────────────────────────────────┐
│         LLM 客户端抽象接口            │
│  chat(messages) → Response           │
│  chat_stream(messages) → Stream      │
└──────────────────────────────────────┘
          ↑           ↑           ↑
    ┌─────┴─────┐ ┌──┴──────┐ ┌──┴──────────┐
    │ OpenAI    │ │Anthropic│ │ Google       │
    │ Compatible│ │         │ │              │
    │ (包括任意 │ │         │ │              │
    │  OpenAI   │ │         │ │              │
    │  协议服务)│ │         │ │              │
    └──────────┘ └─────────┘ └──────────────┘
```

- 抽象接口定义统一的 `chat` / `chat_stream` 方法，屏蔽不同提供商的 API 差异
- 三套实现：OpenAI-compatible（覆盖 OpenAI、DeepSeek、Moonshot 等兼容 OpenAI 协议的服务）、Anthropic、Google
- 用户通过配置文件指定使用的提供商及对应的 API Key / Endpoint

## 6. 用户故事

### US-01 新手删除文件
> 作为不熟悉 Linux 命令的用户，我希望输入"删除所有 tmp 开头的文件"，系统能告诉我该用什么命令、解释命令含义、并警告风险，让我安全地完成操作。命令不涉及交互，我选择"运行"直接执行。

### US-02 开发者更新系统
> 作为开发者，我希望输入"更新系统"后，系统能自动识别我的 Linux 发行版和包管理器，生成正确的更新命令。由于命令需要 sudo，系统判定为交互式命令，展示"运行并接管"选项。我选择运行并接管后进入默认模式——LLM 每次要输入密码或确认时都需要我授权。熟悉后我切换到全自动模式，LLM 自动完成整个更新流程。

### US-03 不满意初版命令
> 作为用户，当 LLM 生成的命令不符合预期时，我希望能够通过"修改"选项输入修改指示让它重新生成，而不是退出 termhelper 再手动重写整个查询。

### US-04 只想复制命令
> 作为有经验的用户，我不需要软件帮我执行命令，只想快速拿到命令复制到剪贴板，自己决定何时以及如何执行。

### US-05 不想让 LLM 接管终端
> 作为谨慎的用户，当生成的命令需要交互（如 sudo）时，我可能不想让 LLM 完全接管我的终端。我选择"仅复制"，将命令复制到剪贴板，在自己的终端中手动完成交互操作。

### US-06 PTY 接管中的安全控制
> 作为关注安全的用户，我希望在 PTY 接管模式下，LLM 每次要向终端写入内容时都先经过我的确认（默认模式）。当我对 LLM 的决策有信心后，可以切换到全自动模式让流程更顺畅。

## 7. 术语表

| 术语 | 定义 |
|------|------|
| 自然语言查询 | 用户在 `??` 后输入的日常语言描述，如"删除所有 std 开头的文件夹" |
| TUI | 终端用户界面，termhelper 以全屏模式接管终端进行交互展示 |
| 诊断命令 | LLM 为调查系统环境而请求执行的探测性命令，如 `cat /etc/os-release` |
| 环境上下文 | 经调查获得的系统环境信息集合，持久化存储供后续复用 |
| 非交互式命令（`interactive: false`） | 执行过程不需要用户交互的命令（如 `rm`、`ls`），可直接 spawn 执行 |
| 交互式命令（`interactive: true`） | 执行过程中需要用户交互的命令（如 `sudo apt upgrade`），必须通过 PTY 接管或降级为复制 |
| PTY 接管 | 通过伪终端让 LLM 监控程序输出并自动应答，模拟人工交互；对应交互式命令的"运行并接管" |
| 默认模式（PTY） | PTY 接管的一级权限：LLM 分析输出并提议写入内容，用户逐次确认后方可写入 |
| 全自动模式（PTY） | PTY 接管的二级权限：用户主动切换后，LLM 完全自主决策无需逐次确认；可随时切回或中断 |
| 普通执行 | spawn 子进程执行命令；仅适用于非交互式命令 |
| Shell RC 命令定义 | 在 Shell RC 文件中写入 alias 或 def，使 `??` 被 Shell 解析为 termhelper 调用 |
| 安全分析 | 对生成命令进行风险评级并产生对应警告提示的过程 |
| 会话 | 一次 `??` 触发到 TUI 退出的完整生命周期；对话历史不跨会话保留 |
